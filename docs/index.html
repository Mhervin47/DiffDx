System deep-dive

Dense feedback for sparse rewards

A correct diagnosis arrives once, at the very end of a conversation that can run fifteen questions deep. DiffDx borrows a trick from reinforcement learning — a separate critic that scores every turn — and builds a full three-loop diagnostic pipeline around it.

DiffDx is a multi-turn diagnostic dialogue system built on an Actor-Critic loop. A doctor model interviews a patient to build a ranked differential; a critic grades every question; a router turns the result into a specialty referral. Here is how every piece works — and why it is shaped the way it is.

01 — the problemSparse rewards, long dialogues

Medical diagnosis is an awkward learning problem. The only unambiguous signal — was the final diagnosis correct? — arrives once, at the end of a conversation that can span fifteen turns. Everything before that is a sequence of questions that were either well-chosen or weren't, and there is no label at the turn level telling you which.

Training a model against that lone terminal reward is possible but wasteful. The reward is sparse (one bit, at the end) and delayed (many actions removed from the outcome). The model has to guess which of its fifteen questions deserves credit for a correct final answer — this is the classic credit-assignment problem, and it gets worse as the episode length grows.

DiffDx sidesteps it by not trying to fine-tune yet. Instead it builds the scaffolding first: a system that runs real diagnostic dialogues and a critic that scores every turn as it happens, producing the dense intermediate feedback that a single terminal reward cannot. The data-collection architecture is the product of Loop 1 and Loop 2. Preference-tuning those critic-scored sessions is the next loop.

02 — the framingActor-Critic, adapted for language

The design maps standard Actor-Critic (A2C) reinforcement learning onto a diagnostic dialogue. In classic A2C, an actor generates actions in an environment, a critic estimates how good those actions are, and the advantage function measures how much better an action was than what the critic expected. The policy improves from that signal.

The translation to language is direct but requires one extension: the environment is a patient — real, or simulated from a structured record. Every session generates up to fifteen (state, action, reward) tuples, one per turn. Those tuples are the eventual training data for DPO fine-tuning. Here is what each component means concretely.

State
PatientProfile
schemas.py
The full knowledge state at the start of each turn. It accumulates across turns and is the document passed to the actor on every call. Fields: demographics, chief_complaint, symptoms (current only, with onset/severity), history (past/chronic: medical, medications, allergies, family, social), ruled_out and ruled_in (diagnoses only — not denied symptoms), free_notes, and running_summary (LLM-generated compression of older turns). The schema is locked: fields can be added, never renamed or removed, so every evaluator written today reads sessions from any prior phase without a migration.
Action
DoctorTurnOutput
schemas.py · doctor.py
The actor's output on each turn — a strict JSON object carrying: turn_index, current_differential (list of {dx, prob} pairs that must update from the prior turn's values), biggest_uncertainty (the single information gap limiting further narrowing), candidate_questions (2–4 options), chosen_question (plain language, one question only — no bundling), rationale (references specific dialogue evidence, not generic phrases), confidence_to_stop (0.0–1.0 cumulative confidence), should_stop (true when confidence ≥ 0.75), and safety_flags (for immediate danger only — not clinical symptoms).
Reward
TurnCritique
critic.py · critic_turn_v0_1.txt
Three 0–1 scores computed by a blind critic after the session. Question quality score — was this the highest-yield question to ask at this moment? Differential quality score — does the updated probability distribution correctly reflect the accumulated evidence? Reasoning quality score — does the rationale reference specific dialogue evidence and name the hypothesis being tested? The critic never sees the patient's answer or the ground-truth diagnosis. This is forward-looking evaluation: it judges the question before the response lands. A great question that led to a wrong answer still scores highly; a lazy one that fluked into the right answer scores low. The composite reward R = mean(Q + D + reasoning) / 3.
Advantage
A(s, a) = R − V(s)
aggregator.py
How much better this particular action was than the expected baseline for the current profile state. A positive advantage means the actor chose a question the critic rated above average for sessions at a similar diagnostic moment; a negative advantage means it underperformed the baseline. Advantage is the scalar that will drive DPO preference-pair construction: turns with high positive advantage become "chosen" examples; turns with negative advantage become "rejected" examples for the same state. The credit-assignment problem that makes terminal rewards hard to learn from is solved here — each turn gets its own advantage, not a shared end-of-episode signal.
Value
V(s)
aggregator.py
The expected cumulative diagnostic quality achievable from the current state — the baseline that advantage is measured against. V(s) is currently estimated from session-level aggregate critic scores across sessions with similar profile states; it is not yet a learned value network. As annotated sessions accumulate, the baseline becomes more precise. For now, turns from a session that scores above the session-mean reward are treated as positive-advantage; turns below are treated as negative-advantage. This is a deliberate simplification — the architecture supports a full learned V(s) when data is sufficient.
Actor
Doctor LLM
doctor.py · doctor_v0_6.txt
The policy that maps a PatientProfile state to a DoctorTurnOutput action. The actor is not fine-tuned — it is a prompted large language model (groq/llama-3.3-70b-versatile by default) guided by a versioned system prompt and three dynamically retrieved exemplars per turn. The prompt enforces a nine-step reasoning chain before any JSON is output. Base-rate calibration, evidence-based differential updating, mandatory diagnostic breadth, and plain-language communication are all hard-coded constraints in the prompt — they are not expected to emerge from the model's pretraining alone. This design lets the architecture collect high-quality training data now; the actor will be fine-tuned on that data in a future loop.
Environment model
DDXPlus simulator
patient_simulator.py
The environment model used for offline evaluation — a simulated patient grounded in a DDXPlus record. Unlike a real patient, the simulator's ground truth is known, enabling automatic accuracy measurement. It answers doctor questions truthfully in natural language, cross-referencing the structured evidence dict before each response to prevent invented symptoms. A leakage scan regenerates any response that accidentally names the ground-truth pathology. A disclosed_symptoms set tracks what has been revealed to prevent contradictions across turns. The simulator is not used during real patient sessions — only during the Loop 2 evaluation pipeline.

Why a turn-level critic

The critic evaluates reasoning quality, not outcomes. It never needs a ground-truth diagnosis to judge whether a question was the highest-yield thing to ask at a given moment — which is exactly the signal a terminal reward cannot supply. A great question that leads to a wrong answer still gets a high critic score; a lazy question that flukes into a correct answer still gets a low one.

Today, the critic runs post-hoc — after a session ends, not during it. The advantage A(s,a) can be computed from the collected scores but is not yet fed back into the actor at inference time. Collecting well-annotated (state, action, reward) tuples at scale is the current work.

04 — the mapThe architecture at a glance

Three decoupled loops feed a web application. Each was built in sequence; later loops read artifacts written by earlier ones — log files, final records, routing decisions. They never call back. That one-directional dependency is what makes each loop independently testable and replaceable.

05 — the engineLoop 1: the doctor turn, step by step

Every question the doctor asks follows the same five-step outer loop. The state entering a turn is a PatientProfile; the output is a strict-JSON DoctorTurnOutput carrying the updated differential, chosen question, rationale, and stop signal. Between turns, a second lighter call — the profile updater — extracts new information from the patient's answer and merges it into the profile before the next turn begins.

  1. Build the retrieval query Concatenate chief_complaint + running_summary + the five highest-severity symptom names into a single string. This query will change turn by turn as the profile evolves.
  2. Retrieve three exemplars The query is embedded with all-MiniLM-L6-v2 (384-dim, on-device), then FAISS + MMR selects three worked clinical examples relevant to the current diagnostic moment. The retrieval re-runs every turn because the profile — and therefore the best exemplars — changes.
  3. Load the versioned doctor prompt Pull prompts/doctor_v0_6.txt from disk. The version is recorded in model_metadata so any session can be replayed against any past prompt version exactly.
  4. Assemble the full prompt System half = the doctor prompt (clinical rules + nine-step reasoning instructions) + the three exemplars injected under an "Exemplar demonstrations" header. User half = the patient profile JSON + compressed turn history. The exemplar section is always labelled clearly to distinguish it from the real patient's data.
  5. Call, parse, validate, retry Call the actor LLM. Strip any markdown fences (some models add them despite instruction). Parse JSON. Validate against the DoctorTurnOutput Pydantic schema. On failure, feed the parse error back to the LLM and retry up to three times. On a rate-limit (429), switch to the next provider immediately — no sleep.

The provider fallback chain

All LLM calls flow through a single interface. On 429 the next provider is tried immediately. On 5xx, exponential backoff via tenacity kicks in. The model that actually answered is always logged in model_metadata so sessions are reproducible and model-swaps are visible in the data.

Termination conditions

The session loop terminates on the first of three events. If confidence_to_stop reaches 0.75, the prompt itself sets should_stop: true and the loop exits cleanly. The session config's confidence_to_stop: 0.55 threshold provides a secondary check: the loop monitors the value each turn and can exit even if the actor hasn't set the stop flag. The hard cap of 15 turns fires regardless. On any emergency phrase in patient input, a safety stop fires immediately — the session terminates, the patient sees an emergency services message, and termination_reason is logged as "safety_stop".

06 — inside the turnThe nine-step reasoning chain

Before the actor writes a single JSON field, the doctor prompt forces a chain-of-thought through nine steps in a fixed order. This is injected as the last section of the system prompt, and the LLM must complete all nine before it is permitted to output the JSON object. The reasoning is visible in the rationale field and in the model's pre-JSON thinking when temperature is above zero.

1
Scan prior turns. Review the patient profile and all turn history. State explicitly what is already known — this prevents the actor from ignoring information already documented.
2
Identify what the latest answer revealed. Name every diagnosis the answer shifts, in which direction, and by how much. Reference the previous turn's differential numbers as the starting point — not the chief complaint.
3
Build the updated differential. Apply the shifts from step 2 to the prior values. Every confirmed finding raises relevant diagnoses by ≥0.10; every denial lowers them by ≥0.10. Before removing any diagnosis, verify the patient specifically denied a pathognomonic finding — otherwise, keep it at a reduced probability.
4
Identify the biggest uncertainty. The single information gap whose absence most limits further narrowing. This becomes the biggest_uncertainty field and drives question selection.
5
Generate 2–4 candidate questions. Each must directly address the biggest uncertainty. Write them in plain patient language before evaluating.
6
Choose the single best question. Sentence case, one question only — no bundling. If the systemic screening rule applies (joint/muscle pain, not yet asked), this question wins regardless of other candidates.
7
Write the rationale. Must reference specific evidence from the dialogue (not generic "will help differentiate"). Must name the hypothesis being tested and why this question beat the others. Differential numbers should appear when they inform the choice.
8
Set confidence_to_stop. Apply the calibration anchors. Apply the decisive closure rule: if the leading diagnosis exceeds 0.65 and the runner-up is below 0.20, commit — set ≥0.75 and should_stop: true. Do not manufacture questions to delay closure.
9
Check safety_flags. Use only for: immediate patient danger, suicidal/homicidal ideation, or a minor presenting without a guardian on a sensitive topic. Never use for clinical symptoms — those belong in the differential. Leave empty in all other cases.

07 — the statePatientProfile and the profile updater

The PatientProfile is the Actor-Critic state — the structured representation of everything known about the patient at a given point in the session. It is the document passed to the doctor actor on every turn. Its schema is locked: fields can be added across versions, but nothing is renamed or removed — so evaluators and critics written today can read sessions from any prior phase.

After every patient answer, a profile updater fires — a lighter LLM call (a different prompt, same model) that extracts a ProfileDelta from the Q&A pair and merges it into the profile. The updater prompt is hard-constrained: "extract only, never infer." Key rules it enforces:

  • Denied symptoms do not create new entries and do not go into ruled_out. Ruled-out is for diagnoses only, not denied symptoms.
  • Before adding a new symptom, check if it already exists under a different label — update severity/onset rather than duplicate.
  • Social history is lifestyle only: smoking, alcohol, occupation, diet, exercise. Current symptoms must never land in history.social.
  • If the answer contains nothing new, return an empty delta — do not invent updates to justify the call.

The separation between the updater and the actor matters: the actor's job is to ask the best question; the updater's job is to extract what the answer revealed. Mixing them would contaminate both.

08 — the expertiseTeaching by example: how ICL works

Before any fine-tuning exists, how does the actor know what a good diagnostic question looks like? Through in-context learning (ICL): hand-curated worked examples injected into the prompt at inference time, every single turn.

ICL works here because the space of "what makes a good diagnostic question" is not easily captured by a generic instruction. A good exemplar shows the model a concrete moment — this profile, this differential, this uncertainty — and demonstrates the reasoning that leads to the highest-yield question. The model generalises the reasoning pattern, not the surface question.

The pool holds roughly fifty hand-curated exemplars, all marked frozen: true. Frozen means they are permanent — they will never be evicted or reweighted even when future loops add machine-generated exemplars. They are the seed set that guarantees baseline coverage of core clinical scenarios.

Three design choices make this exemplar work. First, only context_summary is embedded — retrieval matches on the situation, not the answer. Second, the rationale explains the reasoning structure: why this information gap, why this question beats the alternatives, what confirmation would change. Third, the question is specific enough to be useful as a pattern but general enough not to be copied verbatim — the doctor prompt's exemplar-usage rule forbids literal copying.

The three things that transfer are: how to identify the highest-value information gap, how to evaluate candidate questions against it, and how to justify the choice with reference to specific dialogue evidence.

09 — the selectionFAISS + MMR: how exemplars are retrieved

Retrieval is dynamic — it re-runs at every turn because the profile changes. The same patient presenting chest pain at turn 0 (general presentation) and at turn 4 (confirmed OCP use + pleuritic quality + immobilisation established) is in a very different diagnostic moment and needs different exemplars.

Step 1: embed the query

A query string is assembled from the current profile: chief_complaint + " | " + running_summary + " | " + top_5_symptoms_by_severity. This is embedded by all-MiniLM-L6-v2, a 384-dimensional sentence transformer that runs locally on device — no API call, no latency, no rate-limit exposure. All exemplar embeddings are pre-computed once via embed_exemplars.py and cached in the FAISS index.

Step 2: cosine similarity via FAISS

The FAISS index is a flat inner-product index. Exemplar embeddings are unit-normalised at index-build time, so inner product = cosine similarity. FAISS returns the top-20 most similar exemplars as candidates for the MMR pass.

Step 3: MMR reranking for diversity

Maximal Marginal Relevance selects three exemplars that are relevant to the query and diverse from each other. Without it, FAISS would often return three near-identical exemplars from the same disease cluster. MMR prevents that:

# pick the exemplar that maximises this composite score
score(e) = λ · sim(query, e) − (1 − λ) · max_sim(e, already_selected)

# λ = 0.5 → equal weight on relevance vs. diversity
# already_selected starts empty; each pick updates it
# repeat until k=3 exemplars are selected

With λ = 0.5, an exemplar that is very similar to the query but nearly identical to an already-selected exemplar scores no better than a slightly less relevant but genuinely different one. This guarantees the three injected exemplars cover distinct diagnostic moves rather than repeating the same pattern three times.

Why three, not five?

More exemplars means more tokens, which crowds the patient profile and turn history. Empirically, three exemplars provide enough pattern variety without consuming more than ~15% of the prompt budget. The current mmr_lambda: 0.5 and top_k_exemplars: 3 are in config.yaml and tunable without touching any code.

10 — the rulesThe prompt system

Every prompt template carries a version suffix and is treated as immutable after deployment. Old sessions record their prompt version in model_metadata. A session from three months ago can be replayed exactly by pointing to the same prompt file. You never edit a prompt in place; you copy it to the next version, change it, update the config, and run a bake-off comparison on fixed test cases.

The doctor prompt — nine hard constraints

Beyond the nine-step reasoning chain, doctor_v0_6.txt encodes nine clinical guardrails that fire on every turn:

11 — the budgetKeeping context small

Fifteen turns of raw Q&A plus full profile JSON would exhaust a 4 k-token context in about seven turns. From turn 4 onward, once the history window exceeds the three most-recent turns, DiffDx compresses in two stages. The deterministic stage runs first — this is the critical design choice.

Stage 1 · code-level dedup (no LLM)

Symptom merging by: exact match → substring → content-word subset → Jaccard overlap >80%. History list deduplication by case-insensitive match. This catches the most common failure mode — the same symptom logged under three slightly different labels across three turns — with zero hallucination risk.

Stage 2 · LLM recategorization + summary

Only after dedup does an LLM see the cleaned profile. It fixes miscategorized items (current symptoms that drifted into history.medical) and generates the running_summary: 2–4 sentences covering confirmed symptoms with timeline, key ruled-in/out findings, and open questions. The instruction is "extract only, never infer." The last three turns always pass through verbatim; everything older lives only in the summary.

The sequence matters: the LLM summary step only ever operates on already-deduplicated, already-cleaned input. Running the LLM first would mean asking a generative model to summarize a document containing three copies of the same fact — a reliable path to hallucinated consolidations.

12 — the judgeLoop 2: the critic

Loop 2 reads the JSONL session logs written by Loop 1. It never calls back into the dialogue runtime. For every session, the critic scores each turn independently, and the DDXPlus evaluator checks the final answer against a ground-truth label.

How the critic scores

The critic is blind on two dimensions. It never sees the ground-truth diagnosis — so it cannot cheat by awarding high scores to turns that happened to approach the right answer. And it never sees the patient's actual response — so it scores the question as asked, before the answer lands. This is forward-looking evaluation: was this the highest-yield question to ask at this moment, given only what was known when the question was written?

Why a different model

The critic must be a different model from the actor. Self-scoring is not useful — a model consistently rates its own reasoning style highly regardless of actual quality. The default critic (openrouter/google/gemma-4-31b-it:free) and the doctor (groq/llama-3.3-70b-versatile) are different families. The CRITIC_MODEL environment variable overrides the default without touching code.

Across a session, the aggregator in loop2/critic/aggregator.py rolls up per-turn scores into session-level statistics. Sessions where the actor consistently produced high advantage (R > V(s)) become the "chosen" side of DPO preference pairs in future training. Sessions where the actor underperformed the expected baseline become "rejected" pairs. In Actor-Critic terms, advantage A(s,a) = R − V(s) is the scalar the DPO trainer will use — it measures how much better a given action was than the baseline the value function expected.

13 — the ground truthDDXPlus evaluation

The critic scores reasoning quality — but cannot check whether the final answer is actually right. That is the job of the DDXPlus ground-truth evaluator, which runs in the same Loop 2 pipeline and requires no LLM at all.

What DDXPlus is

DDXPlus is a synthetic medical dataset: 70+ pathologies, approximately 200 symptoms and antecedents, and thousands of patient records. Each record carries:

DDXPlus patient record structure
{
  "patient_id":            "P_0041822",
  "age":                    34,
  "sex":                    "F",
  "initial_evidence":       "E_88",  # chief complaint code
  "evidences": {            # symptom presence / value
    "E_12": true,           # pleuritic chest pain: present
    "E_44": false,          # haemoptysis: absent
    "E_71_@_3": true        # OCP use, coded value
  },
  "antecedents": {          # risk factors / history
    "A_05": true            # prior DVT: present
  },
  "ground_truth_pathology": "Pulmonary embolism"
}

The raw codes are decoded to human-readable text by loader.py using the companion release_conditions.json and release_evidences.json files. DDXPlus is too large to ship in the repo; it is downloaded separately and stored in data/DDXPlus_Raw/ which is gitignored.

The patient simulator

During Loop 2 eval runs, a SimulatedPatient reads a DDXPlus record and answers doctor questions in natural language. The simulator's job is to stay truthful — it can only disclose what the record contains — without ever naming the ground-truth pathology.

  • Grounded answering. Before generating each response, the simulator cross-references the question against the decoded evidence dict. It cannot claim a symptom is present if the record marks it false, and vice versa.
  • Consistency tracking. A disclosed_symptoms set grows with each answer. The simulator checks prior disclosures before responding to prevent contradictions.
  • Leakage detection. After generating each response, the simulator scans the text for the pathology name (exact + common synonyms). If found, the response is discarded and regenerated up to three times. Persistent leakage fails the eval run for that patient and is flagged in the output.
  • Natural language. Responses are written as a patient would speak — not clinical notation. "I've had some swelling in my left leg, it started a couple of days ago" rather than "DVT sign positive."

The ground-truth evaluator

Pure string matching — no LLM, fully deterministic. After a session ends, the evaluator compares the actor's primary_diagnosis against the DDXPlus ground_truth_pathology.

The T2C (turns-to-correct) metric is particularly useful for identifying where exemplars help: in-pool diseases tend to surface in the differential earlier because the actor has seen worked examples from that disease cluster.

The 20-patient eval set

Twenty hand-curated patient IDs are committed to the repo in data/ddxplus_eval_set.json. The set is stratified to stress-test three distinct failure modes:

  • 5 in-exemplar-pool patients. Diseases with at least one frozen exemplar. Expected performance: high. A regression here means exemplar retrieval broke or the actor stopped using exemplars.
  • 10 out-of-pool patients. Diseases not covered in the exemplar pool. Expected performance: lower, but still reasonable — tests the actor's base clinical reasoning without ICL assistance.
  • 5 deliberately ambiguous differentials. Presentations where two or more diseases have similar symptom profiles. Expected performance: top-3 containment matters more than top-1 here. Tests whether the actor keeps multiple hypotheses alive appropriately.

14 — the handoffLoop 3: routing, safety first

Loop 3 reads a FinalRecord written by Loop 1 and outputs a RoutingDecision: a specialty, urgency level, appointment type, and whether the case is ambiguous. All of this comes from a 70+ disease map — no LLM in the main routing path. The logic lives in three rules applied in a fixed order.

Rule 1: safety override (always runs first)

Emergency takes priority regardless of probability

Any emergency-level disease anywhere in the differential — at 10%, 20%, or 5% — becomes the primary_routing output regardless of which diagnosis leads by probability. A 20% pulmonary embolism is surfaced ahead of an 80% musculoskeletal strain. The asymmetric cost of a missed emergency is the justification: a false alarm costs a same-day visit; a missed PE can be fatal.

Rule 2: confidence adjustment

If the session's final_confidence is below 0.40 and the primary routing's base urgency is routine, it is bumped to urgent. A low-confidence routine referral is worse than a slightly over-cautious urgent one — the doctor can downgrade; the patient cannot un-miss an appointment they weren't sent to.

Rule 3: ambiguity detection

If the top specialty's probability-weighted score is less than 1.5× the second specialty's score, is_ambiguous is set to true and the decision returns two or three alternative_routings alongside the primary. This forces the downstream web app — and the doctor — to see the uncertainty rather than hiding it behind a single confident-looking referral.

15 — the surfaceFrom pipeline to product

All three loops are exposed through a FastAPI application — SQLite in development, Postgres in production via DATABASE_URL. The frontend is vanilla HTML, CSS, and JS with no build step. Two roles, two portals, one universal key.

session_id — the universal foreign key

Every artifact in the system hangs off the session_id: the JSONL session log, the final record, the patient's session history entry, the appointment, the DB-stored session report, and the routing decision. Any component that needs to cross-reference data — the doctor portal loading a patient's prior differential, the patient portal showing doctor-prescribed medications — does so through this key. This is the single design decision that makes the portal coherent without a shared database schema.

The anonymous session problem — and how it's solved

Sessions started before the user logs in have no user association. After login, they're invisible in the sessions list. The /api/auth/sessions endpoint solves this by scanning all of the user's appointments for session_id fields and backfilling any that aren't in the user's session list — pulling their data from the stored session reports. This runs on every sessions-list load. The same backfill runs at booking time so newly linked sessions appear immediately.

The doctor's voice brief

When a doctor opens a patient in the portal, the browser's SpeechSynthesis API automatically dictates a pre-consultation brief: patient name, demographics, chief complaint, reported symptoms, predicted top diagnosis, allergies, medications, and past medical history. No API key — it runs fully client-side. Percentages are deliberately omitted from the verbal brief to avoid anchoring bias; the doctor hears the diagnosis name, not a confidence number.

English as the LLM's lingua franca

The design principle is simple: the LLM always sees English. Sarvam does the boundary work. When a patient submits an answer in Tamil, _sarvam_translate() converts it to English before the session passes it to the doctor model. When the model produces the next question, the /api/tts endpoint translates English back to the session language and synthesises audio using Sarvam's bulbul:v2 voices — returning both the translated text and a base64 WAV. The frontend plays the audio and displays the translated text in parallel.

Why this boundary is the right place to translate

Translating at the API boundary — not inside the prompt — means the reasoning model never has to be multilingual. Prompts, exemplars, DDXPlus records, and critic scoring all stay in English. A future swap of the translation layer (a different provider, a local model) touches two functions, not the entire pipeline.

17 — in hindsightDecisions that paid off

ChoiceWhy it mattered
No blocking sleeps on rate limitsOn 429, immediately try the next provider. Sessions stay responsive; the actual model used is always logged. Staggered critic calls (not simultaneous) avoid burst 429s without adding dead time.
Dedup before compressionThe deterministic pass eliminates the most common failure before the LLM ever sees the input. The LLM summary only operates on already-clean data.
Safety over probability in routingAny emergency-level disease routes first, regardless of probability. False negatives for emergencies cost lives; false positives cost a same-day visit.
Frozen exemplar seed setThe 50 hand-curated exemplars are never evicted. Future machine-generated exemplars with frozen: false can be pruned by critic score; the seeds cannot. This guarantees baseline clinical coverage at every future state of the pool.
Additive-only schemasFields can be added; none can be renamed or removed. Every evaluator written today reads day-one sessions without a migration script.
Profile updater as a separate callKeeping extraction (updater) and generation (actor) in separate prompts prevents each from contaminating the other. The actor asks questions; it does not decide what an answer reveals.
session_id as universal FKA single string ties together every artifact across all three loops and the web app. Cross-component joins are O(1) lookups, not schema migrations.

18 — honestlyOpen problems and what's deferred

What exists today is the architecture and data-collection foundation. Several things are deliberately deferred.

  • No fine-tuning yet. (State, action, reward) tuples are being collected per session turn. DPO/LoRA training on high-advantage sessions — the payoff the whole A2C framing is built toward — is the next loop. The data pipeline is the current deliverable.
  • Critic is post-hoc, not in-loop. Scores are computed after sessions end. A real-time quality gate — where a low critic score mid-session triggers a different exemplar injection or a prompt revision — is deferred.
  • Advantage estimation is approximate. V(s) is estimated implicitly from session-level baseline performance, not from a learned value network. A proper advantage function requires enough sessions at similar profile states to establish a baseline — that data is still accumulating.
  • Evaluation is narrow by design. Twenty stratified DDXPlus patients is enough to detect regressions and sanity-check, not to make statistical claims. Synthetic patients are also not real patients.
  • In-memory session state. The dialogue runtime lives in a Python dict. A mid-session server restart loses the conversation. Final records survive (they're DB-persisted), but active sessions do not. A session-state store (Redis, Postgres) is the production path.
  • Sandbox auth and booking. JWT tokens, SQLite, mock booking slots, and no real provider API integration. EHR/FHIR integration, real identity verification, multimodal input, and real scheduling APIs are all future work.

Status: Loops 1–3 complete, the web app in active development, DPO fine-tuning deferred. The harder problems ahead are not missing algorithms — they are credit assignment across turns, squeezing signal from a one-bit terminal reward, and knowing which of these design choices actually survive contact with real evaluation data.

If there is a single idea worth taking from DiffDx, it is the reframing: a long diagnostic conversation is not one decision to be graded at the end. It is fifteen small ones — each asking which question had the highest information value at that specific moment, in that specific profile state. The seven A2C components described in section two name that judgment precisely. The exemplars demonstrate it at inference time. The critic measures it after the fact. Everything else — the compression, the routing, the web app — is scaffolding built to make that measurement possible and, eventually, learnable.
